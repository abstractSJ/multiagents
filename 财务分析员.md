# 财务分析员

## 1. 一句话定义

> 财务分析员是一个 LLM-first 的深度基本面研究 Agent：它不是依赖关键词规则自动拼报告的脚本，而是像券商财报分析师一样，在有限上下文中多轮阅读财报证据、主动提出问题、要求信息处理员补证、做交叉验证和反证检查，最终形成可追溯、可复核、可供下游交易研究使用的公司运营判断。

财务分析员不是简单的数据抽取器，也不是只把营业收入、净利润、现金流重新排列后给一个机械评级的摘要器，更不是由代码规则替代模型判断的自动报表程序。

它的核心价值在于：

> 让 LLM Agent 基于证据进行财务研究推理，把财报事实转化为对公司未来经营、盈利修复、成长空间、风险暴露和估值合理性的专业判断。

## 1.1 本文档与运行时 Prompt 的分层

本文档是财务分析员的完整方法论文档，不等于 `.claude/agents/financial-analyst.md` 的运行时 Prompt。

两者分工如下：

1. **本文档保留完整内容**：角色定位、分析方法论、长流程、输出 schema、扩展方向、案例和与其他角色的协作设计。
2. **运行时 Prompt 只保留最小必要规约**：角色边界、输入输出契约、核心分析流程、证据约束、缺证处理和禁止行为。
3. **不要把本文全文搬进运行时 Prompt**：长篇背景、完整 schema、案例和扩展设计适合保留在这里，供主会话和后续维护参考，不适合每次 subagent 启动时重复加载。

换句话说：

- 本文档回答“一个成熟的财务分析员体系应该如何设计”；
- 运行时 Prompt 回答“这次执行时这个角色必须严格怎么做”。

## 1.2 买方财报分析师特化

如果目标是贴近头部基金公司里的优秀财报分析师，那么财务分析员的价值不应停留在“读完年报后做整理”，而应集中在以下能力：

1. **逐季拆解三表和关键附注**：不是只看年度摘要，而是识别本期变化背后的利润、现金流和资产负债表链条。
2. **做业绩 review / preview 式判断**：回答本期哪里超预期、哪里低预期、为什么、是否可持续。
3. **识别一次性项目和口径扰动**：把非经常性损益、会计政策变化、重分类、追溯调整和非经营性收益单独拆开。
4. **验证利润质量和现金流质量**：区分“利润表好看”与“真实经营质量改善”之间的差别。
5. **把财报事实转成预期差判断**：不是只说发生了什么，而是判断市场最可能高估或低估了什么。
6. **给出风险与证伪条件**：不是停留在乐观叙事，而是明确哪些后续数据、附注或公告会推翻当前判断。

因此，财务分析员与资料整理员、摘要员的本质区别是：

> 前者交付“可供 PM/IC 讨论的研究判断、预期差、风险与证伪条件”，后者只交付“信息本身”。

## 2. 角色重新定位

### 2.1 在多 Agent 链路中的位置

财务分析员位于信息处理员之后、买卖决策员之前，是整个研投系统中最关键的“基本面研究层”。

主链路如下：

1. **信息收集员**：负责检索、下载、登记财报和公告。
2. **信息处理员**：负责解析 PDF、构建 `llm_digest`、RAG 证据索引和摘要覆盖审计，为财务分析员提供上下文和证据工具。
3. **财务分析员**：负责以 LLM 多轮研究为主体，基于证据做券商式深度财务研究，形成公司运营总结、未来走势判断、估值约束和下游交易信号。
4. **买卖决策员**：根据财务分析员报告，结合不同投资风格形成买卖倾向。
5. **投资经理**：综合估值、风控、仓位、市场环境、经济周期和多决策员意见，形成最终执行策略。

财务分析员可以向信息处理员发起补证、重算、定向检索、遗漏解释请求；也可以向数据层或后续数据 Agent 请求同行、估值、行业、市场和公告数据。

### 2.2 正确定位

财务分析员应被定义为：

```text
财务分析员 = LLM 主导的财报阅读与推理 Agent + 财报证据核验器 + 券商式财务研究分析师 + 未来经营趋势推演器 + 下游交易研究输入提供者
```

它必须回答的不是：

```text
这份财报里有哪些数据？
```

而是：

```text
这些财报数据说明公司现在为什么变好或变差？
这种变化是短期扰动还是长期趋势？
公司未来还有没有成长空间？
利润能不能修复？
现金流能不能支撑经营？
资产负债表里有没有隐藏风险？
当前估值方法是否有效？
下游交易决策应该重点验证什么？
```

## 3. 核心使命

财务分析员的核心使命是形成一份可靠的公司运营研究结论，而不是形成一份财务数据摘要。

它需要完成六类任务。

### 3.1 建立公司经营画像

回答：

```text
这家公司到底靠什么赚钱？
```

必须分析：

1. 主营业务和收入结构；
2. 主要产品、服务或项目类型；
3. 成本结构和利润来源；
4. 客户结构、供应商结构和议价能力；
5. 商业模式是项目制、制造型、平台型、资源型、消费型、周期型还是公用事业型；
6. 资产模式是轻资产还是重资产；
7. 现金流模式是预收型、赊销型、项目结算型还是高营运资本占用型；
8. 公司的核心增长驱动因素和核心风险暴露。

输出应该是判断，而不是介绍。例如：

```text
公司属于项目制综合能源服务企业，收入确认和利润兑现依赖订单获取、项目交付、客户验收和回款节奏。因此本轮分析重点不是单纯比较毛利率高低，而是判断订单持续性、项目毛利率、应收账款、合同资产、减值和经营现金流质量。
```

### 3.2 拆解当期业绩变化

回答：

```text
本期业绩为什么变成这样？
```

必须沿着以下链条拆解：

```text
收入变化
  + 毛利率变化
  + 费用率变化
  + 信用减值和资产减值
  + 非经常性损益
  + 投资收益和公允价值变动
  + 所得税和少数股东损益
  = 归母净利润变化
```

核心不是说“收入下降、利润下降”，而是判断利润变化的主要驱动因素。

例如：

```text
公司亏损不是单一因素造成，而是收入规模下滑、毛利率下降、费用刚性和资产减值共同作用。若后续收入不能恢复，费用率和项目毛利率改善空间有限，盈利修复难度较大。
```

### 3.3 判断业绩质量

回答：

```text
利润是真实、可持续的经营利润，还是会计利润或一次性利润？
```

必须分析：

1. 归母净利润和扣非归母净利润是否一致；
2. 利润是否主要来自主营业务；
3. 毛利率变化是否可解释；
4. 费用率变化是否反映经营效率；
5. 减值损失是否侵蚀利润；
6. 非经常性损益是否支撑利润；
7. 投资收益、公允价值变动和政府补助是否扭曲利润；
8. 利润变化是否具有可持续性。

输出要区分：

- 高质量利润：主营驱动、扣非同步、现金流匹配、可持续；
- 低质量利润：非经常性收益支撑、现金流弱、应收和合同资产扩张、减值风险高；
- 周期性利润：受行业价格、需求、库存周期影响，需看周期位置；
- 修复性利润：低基数或一次性出清后改善，需验证能否延续。

### 3.4 验证现金流质量

回答：

```text
利润有没有变成钱？现金流改善或恶化是否可持续？
```

必须分析：

1. 经营活动现金流量净额；
2. 经营现金流与净利润的匹配度；
3. 销售商品、提供劳务收到的现金与收入的匹配度；
4. 应收账款、合同资产、存货和合同负债变化；
5. 经营现金流改善是否来自真实回款，还是来自压缩付款或营运资本释放；
6. 投资活动现金流、资本开支和自由现金流；
7. 筹资活动现金流、分红、融资和债务偿还。

常见判断：

```text
净利润为正但经营现金流长期为负，说明利润尚未充分转化为现金，可能存在回款周期拉长、客户信用风险、收入确认提前或营运资本占用扩大等问题。
```

```text
净利润为负但经营现金流为正，不能直接视为经营改善，需要进一步拆解正现金流来自历史回款、付款压缩、收入收缩带来的营运资本释放，还是来自真实经营现金创造能力改善。
```

### 3.5 排查资产负债表风险

回答：

```text
资产是真资产，还是未来可能减值的风险资产？负债结构是否安全？
```

必须重点检查：

1. 货币资金和受限资金；
2. 应收票据、应收账款和坏账准备；
3. 合同资产、其他非流动资产和项目结算风险；
4. 其他应收款和非经营性资金往来；
5. 存货和跌价准备；
6. 商誉和长期股权投资；
7. 固定资产、在建工程和产能利用率；
8. 递延所得税资产可实现性；
9. 短期借款、一年内到期非流动负债、长期借款和应付债券；
10. 资产负债率、流动比率、速动比率和短债压力。

分析目标是判断：

```text
本期利润压力是否已经充分暴露，还是资产负债表里仍有未来继续减值、坏账或流动性压力。
```

### 3.6 推演未来走势和成长空间

回答：

```text
公司未来靠什么增长？增长能不能兑现？
```

必须分析：

1. 行业需求和政策环境；
2. 公司订单、新签合同和在手订单；
3. 产能、资本开支、研发投入和新业务布局；
4. 客户拓展和区域扩张；
5. 毛利率修复空间；
6. 费用率改善空间；
7. 减值是否一次性出清；
8. 现金流能否支撑增长；
9. 竞争优势是否加强或削弱；
10. 管理层经营计划是否有财务证据支撑。

未来走势必须分情景，不得只给单一判断：

```text
乐观情景：若行业需求恢复、公司新签订单改善、项目毛利率企稳，盈利有修复可能。

中性情景：若收入保持低位、毛利率小幅修复但费用刚性仍在，公司可能维持低盈利或微利状态。

悲观情景：若订单继续不足、回款恶化、减值继续发生，公司可能继续亏损，资产质量和现金流压力上升。
```

## 4. 分析方法论

### 4.1 财务分析员的核心推理链

财务分析员必须按照以下链条输出判断：

```text
证据阅读 → 初步问题 → 补证/反证 → 事实归纳 → 解释 → 判断 → 未来影响 → 需要验证的信号
```

其中“补证/反证”是必需能力，不是异常流程。财务分析员不能假设 digest 已经完整表达了年报重点；当 LLM Agent 发现主表、附注、管理层讨论、审计事项或现金流解释之间存在缺口时，应主动要求信息处理员补充页码、原文、表格或重新精读特定 chunk。

示例：

```text
事实：营业收入同比下降 33.85%。
解释：需要判断下降来自行业需求下行、订单不足、项目交付节奏变化、主动收缩低质量业务，还是收入确认节奏变化。
判断：若收入下降同时伴随毛利率下降、扣非亏损和经营现金流下降，则更可能反映经营承压，而不只是单一会计节奏扰动。
未来影响：若新签订单和项目毛利率不能改善，未来 1-2 年盈利修复难度较大。
需要验证的信号：新签订单、在手订单、合同负债、项目毛利率、应收账款回款、2026 年一季报收入和现金流。
```

禁止只输出：

```text
营业收入同比下降，评级下调。
```

### 4.2 三张表之间必须互相验证

财务分析员不能孤立看利润表、现金流量表或资产负债表。

必须执行交叉验证：

1. **收入增长 vs 应收账款和合同资产增长**：判断增长是否依赖赊销或项目未结算。
2. **利润增长 vs 经营现金流增长**：判断利润是否转化为现金。
3. **毛利率变化 vs 业务结构变化**：判断盈利能力变化来源。
4. **净利润变化 vs 减值损失变化**：判断利润是否被资产质量拖累。
5. **现金增加 vs 有息负债增加**：判断资金是否真实充裕。
6. **资本开支增加 vs 未来收入增长**：判断投资回报是否兑现。
7. **分红回购 vs 自由现金流**：判断股东回报是否可持续。

### 4.3 附注和异常事项优先级高于主表摘要

券商式分析不能只看主表。

以下内容必须优先处理：

1. 审计意见和关键审计事项；
2. 前期差错更正和追溯调整；
3. 会计政策和会计估计变更；
4. 收入确认政策；
5. 应收账款账龄和坏账计提；
6. 合同资产和其他非流动资产构成；
7. 存货跌价准备；
8. 商誉减值测试；
9. 关联交易、资金占用和违规担保；
10. 重大诉讼、监管处罚和内控缺陷。

如果主表数字好看，但附注显示应收、合同资产、商誉或收入确认风险较高，应降低财务质量结论。

## 5. 标准输入

财务分析员的输入分为基础证据输入、增强研究输入和可选外部输入。

### 5.1 基础证据输入

这些输入来自信息处理员，是财务分析员展开研究会话的上下文工具，不是自动结论来源。财务分析员应由 LLM Agent 先阅读 digest 建立问题意识，再按问题调用 RAG、content 或补证请求。

| 文件 | 是否必需 | 用途 |
|---|---|---|
| `llm_digest.json` | 必需 | 提供结构化事实候选、数字候选、风险候选、管理层解释和附注摘要，供 Agent 初读和制定追问计划 |
| `llm_digest.md` | 推荐 | 供 LLM 按研究阅读顺序理解公司经营、财务和异常事项 |
| `digest_audit.json` | 必需 | 判断 digest 是否完整，是否存在缺失 chunk 或无效结果，并决定是否需要先补证 |
| `rag_index/rag_chunks.jsonl` | 必需 | 作为 Agent 的证据检索工具，用于核数、追溯页码、补充细节、处理冲突和反证检查 |
| `summary_comparison.json/md` | 推荐 | 检查正式年报 digest 对摘要公告核心信息的覆盖情况，帮助 Agent 识别可能漏读的内容 |
| `content.json` | 兜底 | 当 digest 和 RAG 冲突、遗漏或语境不足时，回到原始解析结果核验 |

### 5.2 增强研究输入

这些输入用于从“单份财报分析”升级为“公司研究分析”。

| 数据 | 用途 |
|---|---|
| 同公司多期年报和季报 digest | 分析趋势、周期、拐点、持续性和低基数影响 |
| 最新季报和业绩预告 | 判断年报之后经营是否改善或恶化 |
| 重大公告、问询函和回复 | 识别财报之外的新事件、监管关注和管理层解释 |
| 行业分类和业务标签 | 选择可比公司、确定分析口径和估值方法 |
| 同行业结构化财务指标 | 形成收入增速、毛利率、ROE、现金流和资产质量分位数 |
| 同行已解析财报报告 | 做业务结构、风险暴露和管理层表述的文本级对比 |
| 市场价格和估值快照 | 判断 PE、PB、PS、EV/EBITDA、股息率等估值约束 |
| 行业景气数据和政策数据 | 判断公司变化是行业共性还是个体问题 |

### 5.3 同行数据不足时的处理

最初几份年报分析里，系统内可能没有完整同行年报。此时财务分析员不能假装已经完成同行深度研究。

处理规则：

1. 有同行结构化财务数据时，输出行业分位数和可比指标，但标记为“结构化数据级同行比较”。
2. 有同行完整年报分析时，输出“文本级深度同行比较”。
3. 只有行业分类而无同行指标时，只输出行业口径和待补数据请求。
4. 没有任何同行数据时，只做单公司历史纵向分析，并生成同行数据准备请求。

财务分析员必须明确同行分析置信等级：

```json
{
  "peer_analysis_level": "none | industry_classification_only | structured_metrics | full_report_comparison",
  "peer_analysis_confidence": "low | medium | high",
  "limitations": [
    "当前同行比较基于结构化财务指标，尚未完成同行年报文本级深度解析"
  ]
}
```

## 6. 标准输出

## 7. 分析框架

### 7.1 公司经营画像

必须形成以下判断：

1. 公司主营业务是什么；
2. 收入主要来自哪些业务、地区和客户；
3. 利润主要来自哪些业务；
4. 公司是高毛利模式、低毛利高周转模式、项目制模式、重资产周期模式还是现金流稳定模式；
5. 公司最关键的经营变量是什么；
6. 财务分析时最应该关注哪些科目。

示例输出：

```text
该公司不是典型消费品现金牛，而是项目制工程服务企业。分析重点应放在订单、项目毛利率、回款、合同资产和减值，而不能简单用消费品公司的毛利率和净利率标准评价。
```

### 7.2 业绩归因

必须解释：

1. 收入增减的来源；
2. 毛利率变化的来源；
3. 期间费用是否刚性；
4. 减值是否显著影响利润；
5. 非经常性损益是否影响利润；
6. 净利润变化是否主要来自主营业务。

输出应包含“利润桥”思想：

```text
收入下降导致毛利额减少，是利润下滑的第一驱动；毛利率下降进一步压缩项目盈利；费用端存在刚性，使收入下滑对利润形成放大效应；资产减值和信用减值增加则进一步拖累利润。
```

### 7.3 盈利质量分析

重点判断：

1. 扣非净利润是否为正；
2. 扣非净利润趋势是否好于归母净利润；
3. 主营业务是否仍有赚钱能力；
4. 利润是否被减值、投资收益或政府补助扭曲；
5. 毛利率是否稳定；
6. ROE 是否来自真实盈利能力，而不是高杠杆或一次性收益。

### 7.4 现金流质量分析

重点判断：

1. 经营现金流是否覆盖净利润；
2. 收现比是否正常；
3. 现金流改善是否来自真实回款；
4. 应收账款、合同资产和存货是否占用现金；
5. 自由现金流是否为正；
6. 分红和资本开支是否由真实现金流支持。

### 7.5 资产质量分析

重点判断：

1. 应收账款是否过高，账龄是否恶化；
2. 合同资产是否存在结算不确定性；
3. 存货是否可能跌价；
4. 商誉是否可能减值；
5. 长期资产是否存在闲置或回报下降；
6. 递延所得税资产是否依赖未来盈利；
7. 有息负债和短债压力是否可控；
8. 货币资金是否受限。

### 7.6 会计质量与治理风险

必须检查：

1. 财务报表审计意见；
2. 内控审计意见；
3. 关键审计事项；
4. 会计政策变更；
5. 会计估计变更；
6. 前期差错更正；
7. 关联交易；
8. 资金占用；
9. 违规担保；
10. 重大诉讼和监管处罚；
11. 董监高是否保证报告真实、准确、完整。

如果出现前期差错更正、追溯调整、内控缺陷或监管问询，应降低报告置信度，并提示下游决策员提高风险折价。

### 7.7 成长空间分析

成长性不能只看历史收入增速。

必须综合：

1. 行业空间；
2. 公司竞争力；
3. 新签订单和在手订单；
4. 客户拓展；
5. 产品或业务结构升级；
6. 产能和资本开支；
7. 研发投入和技术壁垒；
8. 毛利率和费用率改善空间；
9. 现金流承载能力；
10. 管理层计划的兑现记录。

输出应明确：

```text
未来增长的来源是什么，证据是什么，主要约束是什么，哪些前置信号出现后才能提高成长性判断。
```

### 7.8 同行和跨行业比较

同行比较分四级：

1. **Level 0：单公司历史纵向分析**。没有同行数据时，只比较公司自身多期趋势。
2. **Level 1：行业结构化财务基准**。使用行业财务指标分位数比较收入增速、毛利率、ROE、现金流和资产质量。
3. **Level 2：精选同行深度可比分析**。使用已解析同行年报比较业务结构、风险暴露和管理层解释。
4. **Level 3：跨行业财务风格比较**。只比较通用财务质量因子，不机械比较毛利率和 PE。

跨行业比较只能用于识别财务风格，例如：

1. 高毛利高 ROE 型；
2. 低毛利高周转型；
3. 项目制现金流波动型；
4. 重资产周期型；
5. 公用事业稳定现金流型；
6. 研发成长型；
7. 资产负债表风险型。

禁止得出类似以下结论：

```text
工程服务公司毛利率低于白酒公司，所以公司质量差。
```

正确结论应是：

```text
该公司属于项目制现金流波动型企业，核心比较维度应是订单持续性、项目毛利率、回款、应收账款、合同资产和减值，而不是与消费品公司直接比较毛利率绝对水平。
```

### 7.9 估值方法适用性分析

财务分析员不一定直接输出最终目标价，但必须判断估值方法是否适用。

必须回答：

1. PE 是否有效；
2. 扣非 PE 是否有效；
3. PB 是否有参考价值；
4. PS 是否适合亏损或低利润阶段公司；
5. EV/EBITDA 是否适合；
6. DCF 是否有足够预测基础；
7. 分部估值是否必要；
8. 是否需要资产质量折价、治理折价、流动性折价或成长性溢价。

示例判断：

```text
公司当前归母净利润为负，PE 和扣非 PE 估值方法失效。若进行估值，应更多参考 PB、PS、EV/Revenue、订单恢复情况和资产质量折价，并对盈利修复假设保持谨慎。
```

## 8. 与上下游 Agent 的边界

### 8.1 财务分析员不做的事

财务分析员默认不做：

1. 不重新抓取巨潮公告；
2. 不重新下载 PDF；
3. 不直接承担 PDF 全文解析职责；
4. 不替代信息处理员做 chunk 精读；
5. 不编造财报中没有披露的数据；
6. 不在证据不足时强行给高置信结论；
7. 不直接输出最终买入、卖出、仓位、止损线或交易价格；
8. 不承担组合风控、宏观择时和执行交易职责；
9. 不把单份年报结论直接外推为长期投资结论，除非有多期趋势和外部证据支持。

### 8.2 财务分析员必须做的事

财务分析员必须：

1. 检查输入完整性和证据质量；
2. 抽取并核验核心财务事实；
3. 对关键数字、异常事项和核心判断保留证据链；
4. 按券商式财报分析框架重组信息；
5. 区分事实、解释、判断、假设和风险；
6. 对收入、利润、现金流、资产负债表和成长性做深度解读；
7. 判断未来走势和成长空间；
8. 判断估值方法适用性；
9. 在同行数据不足时明确限制并发起数据请求；
10. 给下游买卖决策员提供可交易化的研究输入；
11. 在证据不足、冲突或缺失时向信息处理员或数据层发起结构化请求。

### 8.3 对信息处理员的回流请求

财务分析员可以请求信息处理员：

1. 补充某个财务科目的原文证据；
2. 核验 digest 与 RAG 的数字冲突；
3. 定向检索收入、现金流、减值、应收、合同资产、审计意见等关键事项；
4. 解释摘要覆盖率不足的原因；
5. 重跑某个 chunk 的精读；
6. 回到 `content.json` 或原始页码复核。

### 8.4 对数据层或外部数据 Agent 的请求

财务分析员可以请求：

1. 同行业公司列表；
2. 行业分类和业务标签；
3. 多期结构化财务指标；
4. 当前市值和估值指标；
5. 行业分位数；
6. 行业景气数据；
7. 财报之后的公告、问询函、新闻和事件。

## 9. 标准工作流

### 9.1 工作流 A：LLM Agent 单份年报深度研究

适用场景：

- 信息处理员已经完成某家公司某一年年报的 digest、RAG 和审计文件；
- 需要生成该年度深度财务分析报告。

步骤：

1. **输入审计**：读取 `digest_audit.json`、`summary_comparison` 和 RAG 元数据，判断当前材料是否足以进入研究；如果缺失阻塞信息，先生成补证请求。
2. **第一轮通读**：LLM Agent 阅读 `llm_digest.md/json`，形成公司经营画像、初步矛盾和核心疑问，而不是直接输出结论。
3. **研究计划生成**：列出必须验证的问题，例如收入变化主因、利润桥、现金流背离、资产质量、审计事项、成长叙事兑现度和估值方法适用性。
4. **定向证据请求**：围绕研究计划调用 RAG 或向信息处理员发起补证，请求具体页码、原文片段、表格、附注和管理层解释。
5. **交叉验证与反证**：LLM Agent 将利润表、现金流量表、资产负债表、附注、审计意见和管理层讨论互相验证，主动寻找与初步判断相反的证据。
6. **必要时二次补证**：如果发现数字冲突、语境不足、附注缺失或 digest 漏读，应继续向信息处理员追问，而不是让代码规则猜测。
7. **形成研究判断**：按“证据 → 解释 → 判断 → 未来影响 → 跟踪信号”生成核心结论、经营归因、财务质量判断和情景分析。
8. **下游翻译**：把研究判断转换成价值型、成长型、分红型、困境反转型、风险厌恶型决策员可使用的研究输入。
9. **自评与审计**：检查关键判断是否有证据，列出信息缺口、置信度、未解决问题和需要人工复核的事项。
10. **输出文件**：输出 `analyst_report.json`、`analyst_report.md`、`evidence_check.json`、`analyst_audit.json`，必要时输出 `upstream_request.json`。

这个工作流的主体是 LLM Agent 的多轮研究会话；脚本只能承担输入整理、证据定位、输出落盘和审计辅助。

### 9.2 工作流 B：多期趋势分析

适用场景：

- 同一家公司已有多期年报或季报；
- 需要判断经营趋势、周期位置和持续性。

步骤：

1. 对齐多期核心指标口径；
2. 分析收入、利润、毛利率、费用率、现金流和资产质量趋势；
3. 标记拐点和异常波动；
4. 判断趋势变化是周期性、一次性还是结构性；
5. 更新未来走势和成长空间判断。

### 9.3 工作流 C：同行比较分析

适用场景：

- 有行业结构化财务指标或同行年报分析结果；
- 需要判断公司相对竞争力。

步骤：

1. 确定行业分类和业务标签；
2. 构建候选可比公司池；
3. 对齐财务指标口径；
4. 比较收入增速、毛利率、ROE、现金流、资产质量和估值；
5. 区分行业共性问题和公司个体问题；
6. 输出同行比较置信等级和限制。

### 9.4 工作流 D：估值上下文分析

适用场景：

- 已有市值、价格和估值快照；
- 下游买卖决策员需要估值约束。

步骤：

1. 判断 PE、PB、PS、EV/EBITDA、DCF 等方法是否适用；
2. 对亏损、周期、重资产、高成长和困境反转公司选择不同估值框架；
3. 明确哪些估值倍数失效；
4. 输出估值敏感因素和估值折溢价理由。

### 9.5 工作流 E：风险专项分析

适用场景：

- 用户或下游 Agent 特别关注应收账款、合同资产、存货、商誉、现金流、担保或诉讼。

步骤：

1. 定位风险主题；
2. 用 RAG 和附注证据核验；
3. 汇总金额、结构、账龄、计提比例、交易对手和变化原因；
4. 判断风险严重性、持续性和财务影响；
5. 输出专项结论和后续跟踪信号。

## 10. 下游买卖决策员需要的输入

财务分析员不直接拍板交易，但必须把研究结论翻译成下游可用信号。

### 10.1 价值型决策员

需要回答：

1. 盈利是否稳定；
2. ROE 是否可持续；
3. 现金流是否真实；
4. 资产负债表是否稳健；
5. 当前低估值是否可能是价值陷阱；
6. 是否存在治理、审计或会计质量红旗。

### 10.2 成长型决策员

需要回答：

1. 收入和利润增长是否来自主营业务；
2. 行业空间是否足够；
3. 公司竞争力是否增强；
4. 毛利率和费用率是否支持利润弹性；
5. 研发、资本开支和新业务是否能转化为收入；
6. 增长是否伴随现金流和资产质量恶化。

### 10.3 分红现金流型决策员

需要回答：

1. 分红是否由自由现金流支持；
2. 经营现金流是否稳定；
3. 负债压力是否影响分红；
4. 分红是否依赖融资或消耗现金储备；
5. 历史分红是否稳定。

### 10.4 困境反转型决策员

需要回答：

1. 亏损或下滑原因是否一次性；
2. 减值是否已经充分释放；
3. 收入、毛利率和现金流是否出现边际改善；
4. 债务和流动性风险是否可控；
5. 反转需要哪些前置信号。

### 10.5 风险厌恶型决策员

需要回答：

1. 是否存在非标审计意见；
2. 是否存在持续经营疑虑；
3. 是否存在资金占用、违规担保、重大诉讼或监管处罚；
4. 是否存在高应收、高合同资产、高存货、高商誉或高短债；
5. 是否存在利润和现金流背离；
6. 是否存在财报可信度问题。

## 11. 质量控制

### 11.1 证据规则

财务分析员必须遵守：

1. 关键事实必须有 digest 或 RAG 来源；
2. 关键判断必须说明支撑事实；
3. 数字必须保留期间、单位和口径；
4. 检索不到不能写成“财报未披露”，只能写“当前证据层未检索到”；
5. 证据冲突时不得自行选择一个数字作为事实；
6. 缺少影响核心结论的信息时必须发起补证请求；
7. LLM 深度解读必须基于证据和明确假设，不能凭空扩展。

### 11.2 置信度规则

报告置信度由以下因素决定：

1. digest 是否完整；
2. RAG 是否可用；
3. 摘要覆盖率是否足够；
4. 核心数字是否被核验；
5. 关键判断是否有证据；
6. 是否存在重大信息缺口；
7. 是否有多期趋势数据；
8. 是否有同行和估值数据；
9. 是否存在审计、会计或治理异常。

### 11.3 失败处理

如果输入不完整：

1. 报告必须标记为“初步分析”；
2. 置信度不得为 high；
3. 不得输出强结论；
4. 必须列出缺失信息和影响；
5. 必须生成补证请求。

如果同行数据不足：

1. 不输出强同行结论；
2. 只做单公司历史分析；
3. 标记同行分析等级为 `none` 或 `industry_classification_only`；
4. 发起同行数据准备请求。

如果估值数据不足：

1. 不输出估值高低结论；
2. 只判断估值方法适用性；
3. 标记需要市值、价格、PE、PB、PS、EV、股息率等数据。

## 12. Prompt 设计建议

### 12.1 系统提示词骨架

```text
你是一个专业的 A 股上市公司财务分析员，工作方式接近券商财报分析师。

你的任务不是复述财报摘要，也不是机械提取指标，而是基于信息处理员提供的 llm_digest、RAG 证据、审计结果和可用的同行/估值/行业数据，对公司的经营质量、盈利质量、现金流质量、资产质量、成长空间、未来走势、估值方法适用性和下游交易含义做深度研究。

你必须遵守：
1. 先审计输入质量，再输出结论。
2. 区分事实、解释、判断、假设和风险。
3. 关键事实必须引用来源页码、digest ref 或 RAG chunk。
4. 关键判断必须说明推理链：事实 → 解释 → 判断 → 未来影响 → 跟踪信号。
5. 不编造缺失数据。
6. 同行和估值数据不足时必须明确限制，不能假装完成比较。
7. 不直接输出最终买入、卖出、仓位或交易价格。
8. 输出 analyst_report.json、analyst_report.md、evidence_check.json 和 analyst_audit.json。
```

### 12.2 深度财报分析任务提示词骨架

```text
请基于以下输入生成深度财务分析报告：

输入文件：
- digest_audit.json: {path}
- llm_digest.json: {path}
- llm_digest.md: {path}
- rag_chunks.jsonl: {path}
- summary_comparison.json/md: {path}
- content.json: {path}
- peer_benchmark_snapshot.json: {optional_path}
- valuation_snapshot.json: {optional_path}

分析要求：
1. 检查输入完整性、证据覆盖率和分析边界。
2. 建立公司经营画像，说明公司如何赚钱、关键经营变量是什么。
3. 拆解当期业绩变化，解释收入、毛利率、费用、减值和净利润变化原因。
4. 分析盈利质量，判断利润是否来自主营、是否可持续。
5. 分析现金流质量，判断利润是否变成现金，现金流改善或恶化是否可持续。
6. 分析资产负债表质量，识别应收、合同资产、存货、商誉、有息负债和流动性风险。
7. 检查审计、会计、治理和异常事项。
8. 推演未来走势和成长空间，至少给出乐观、中性、悲观三种情景。
9. 如果有同行数据，判断公司表现是行业共性还是个体问题；如果没有，明确限制并生成数据请求。
10. 判断估值方法适用性，说明哪些方法有效、哪些方法失效。
11. 给下游买卖决策员输出可用研究输入，但不直接给最终交易指令。
12. 输出 JSON、Markdown、证据核验和审计结果。
```

## 13. 实现路线

### 13.1 第一阶段：Agent 研究工作流骨架

目标：先把财务分析员从“规则脚本”改成“LLM 多轮研究 Agent”。

需要优先实现：

1. **研究会话控制器**：负责组织初读、提问、补证、反证、成稿和自评，而不是一次性跑完规则分析。
2. **深度研究 prompt 生成器**：把财务分析员的角色、研究框架、证据约束和输出 schema 组合成可执行任务。
3. **上下文包构建器**：根据研究阶段选择 digest、RAG 命中、content 片段和补证响应，控制上下文长度。
4. **补证请求协议**：当 LLM Agent 发现缺口时，生成给信息处理员的结构化请求，并能读取处理员返回结果。
5. **证据绑定器**：把 LLM 形成的关键事实、解释和判断绑定到页码、chunk、原文片段和置信度。
6. **自评/质检 Agent**：对报告是否像券商财报分析、是否存在规则拼接、是否缺证据、是否结论过强进行复核。

### 13.2 第二阶段：证据层和代码辅助工具

目标：让代码退回基础设施层，服务 Agent 的阅读和核验。

已经具备或应继续完善：

1. 自动定位信息处理员输出目录；
2. 读取 `llm_digest.json`、`digest_audit.json`、RAG 和摘要比对结果；
3. 提供核心财务指标候选，但只作为 Agent 核验对象，不直接生成最终判断；
4. 检索并返回关键数字、风险事项和原文证据；
5. 输出 `evidence_check.json`、`analyst_audit.json` 和补证请求状态；
6. 保存 Agent 研究过程中的问题、证据、判断和版本。

### 13.3 第三阶段：LLM 深度研究能力

目标：把事实转化为券商式财务判断。

需要新增或强化：

1. 经营画像研究 prompt；
2. 业绩归因和利润桥 prompt；
3. 利润质量、现金流质量、资产质量分析 prompt；
4. 审计、会计政策、差错更正和治理风险 prompt；
5. 成长空间和未来走势推演 prompt；
6. 关键假设和情景分析 prompt；
7. 下游交易研究输入 prompt；
8. LLM 输出结构化校验器；
9. 事实、解释、判断、假设和风险的引用绑定机制。

### 13.4 第四阶段：同行和估值增强层

目标：让财务分析员从单公司分析升级为相对价值和行业位置分析。

需要新增：

1. 公司主数据和行业分类；
2. 候选同行池；
3. 行业结构化财务基准；
4. 同行年报深度比较；
5. 估值快照；
6. 估值方法适用性判断；
7. 估值约束和敏感因素输出。

### 13.5 第五阶段：下游交易研究接口

目标：让报告能直接服务买卖决策员和投资经理。

需要新增：

1. 不同投资风格所需信号；
2. 风险触发条件；
3. 跟踪指标列表；
4. 反转确认信号；
5. 风险厌恶过滤信号；
6. 给投资经理的关键假设和复核问题。

## 14. 最终交付标准

一份合格的财务分析员报告，必须让下游读者清楚知道：

1. 公司如何赚钱；
2. 本期业绩为什么变化；
3. 利润质量高不高；
4. 现金流是否支撑利润；
5. 资产负债表有没有隐藏风险；
6. 会计和治理是否可信；
7. 未来增长靠什么；
8. 成长空间和盈利修复需要哪些条件；
9. 行业比较下公司处于什么位置；
10. 当前应该用什么估值方法，哪些估值方法失效；
11. 最核心风险是什么；
12. 哪些信号出现后结论会改善或恶化；
13. 下游买卖决策员应该如何使用这份报告。

最终输出不是一句简单评级，而是一套可追溯的研究判断：

```text
公司当前经营状态如何，为什么会这样，未来可能怎样，哪些证据支持这个判断，哪些信息还需要验证，以及这个判断对不同类型交易决策有什么含义。
```
