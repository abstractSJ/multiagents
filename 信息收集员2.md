# 信息收集员2

## 1. 一句话定义

> 信息收集员2是专门服务行业研究员的“行业研究输入包收集代理”：它不重复承担财报 PDF 下载职责，也不直接形成行业结论，而是把公司画像、财务分析员输出、业务分部、行业分类、同行候选、产业链、行业指标、政策与新闻指针、市场数据缺口和证据来源，整理成行业研究员可以直接消费的结构化输入包。

信息收集员2不是原信息收集员的替代品，而是并行的数据准备角色。

它的核心价值在于：

> 把“财报资料层”进一步扩展为“行业研究资料层”，让行业研究员能够基于可追溯、可审计、可复用的数据包判断目标公司所在行业、竞争格局、产业链位置和需要补充验证的行业变量。

---

## 2. 角色定位

### 2.1 在多 Agent 链路中的位置

信息收集员2位于原信息收集员、信息处理员、财务分析员之后，行业研究员之前。

主链路如下：

1. **信息收集员**：负责巨潮资讯财报公告检索、PDF 下载、总清单、重放和覆盖审计。
2. **信息处理员**：负责解析财报 PDF，生成 `content.json`、`llm_digest`、RAG 和摘要比对结果。
3. **财务分析员**：负责从财报证据中形成公司经营、盈利、现金流、资产质量和风险分析。
4. **信息收集员2**：负责把行业研究员所需的公司画像、财务摘要、业务结构、行业分类、同行候选、产业链和行业指标指针打包成行业研究输入包。
5. **行业研究员**：基于信息收集员2输出的信息包，判断行业生命周期、景气度、供需、竞争格局、公司行业位置和行业风险。
6. **估值分析员、投资假设生成员、反方观点审查员、风控 Agent、投资经理**：消费行业研究员输出。

### 2.2 正确定位

```text
信息收集员2 = 行业研究资料收集代理 + 本地研究产物汇聚器 + 行业 seed / 外部数据接入器 + 证据来源登记器 + 行业研究员输入包生成器
```

它必须回答的是：

```text
为了让行业研究员分析这家公司，需要哪些行业资料？
这些资料当前哪些已经从本地财报、财务分析员报告和 seed 数据中得到？
哪些资料缺失，必须后续补采？
每条资料来自哪里、可靠性如何、限制条件是什么？
```

它不能回答的是：

```text
这个行业是否值得投资？
这家公司是否应该买入？
估值是否便宜？
目标价是多少？
```

## 3. 与原信息收集员的边界

### 3.1 原信息收集员负责的事

原信息收集员继续负责：

1. 检索巨潮资讯财报公告；
2. 下载和命名财报 PDF；
3. 维护财报总 JSON / CSV 清单；
4. 记录公告编号、PDF 来源、本地路径和下载状态；
5. 基于 manifest 重放下载；
6. 审计财报覆盖情况。

### 3.2 信息收集员2负责的事

信息收集员2默认消费原信息收集员、信息处理员和财务分析员的已有产物。

它负责：

1. 读取财报 manifest，引用原始财报来源；
2. 读取 `content.json`，提取公司基础资料、业务分部和年报元数据；
3. 读取 `analyst_report.json`，提取财务分析员已经形成的财务摘要和经营画像；
4. 读取行业 seed 数据或后续外部行业数据；
5. 生成行业研究员标准输入包；
6. 生成证据表和收集审计文件；
7. 维护行业信息包总 manifest；
8. 明确列出缺失数据和下一步补采请求。

### 3.3 默认不越界

信息收集员2默认不做：

1. 不重新下载财报 PDF；
2. 不替代信息处理员解析 PDF；
3. 不替代财务分析员做财务质量判断；
4. 不替代行业研究员形成行业结论；
5. 不替代估值分析员判断估值高低；
6. 不输出买入、卖出、仓位、目标价或交易建议；
7. 不伪造实时行情、估值、批价、库存、新闻或政策数据；
8. 不把离线 seed 数据包装成权威实时数据。

## 4. 核心使命

信息收集员2的核心使命是为行业研究员提供“可直接研究、可追溯、可审计、可补采”的行业研究输入包。

它需要完成七类任务。

### 4.1 汇聚本地研究产物

优先读取本地已有资料：

1. 财报 PDF manifest；
2. PDF 解析结果 `content.json`；
3. 财务分析员报告 `analyst_report.json`；
4. 财务分析员证据核验和审计结果；
5. 行业 seed 数据；
6. 后续外部行业数据库或公开资料。

### 4.2 建立公司基础画像

输出行业研究员所需的基础公司信息：

1. 公司名称；
2. 股票代码；
3. 交易所；
4. 国家或地区；
5. 报告期；
6. 主营业务；
7. 经营模式；
8. 收入结构；
9. 业务分部；
10. 地区分部。

### 4.3 准备财务摘要输入

信息收集员2不做财务分析，但需要把财务分析员的结果整理给行业研究员使用。

至少包括：

1. 营业收入；
2. 归母净利润；
3. 扣非归母净利润；
4. 经营现金流；
5. 毛利率或业务分部毛利率；
6. 主要财务风险；
7. 财务分析员给出的限制条件。

### 4.4 准备行业分类和同行候选

信息收集员2必须给出行业研究员的初始研究口径：

1. 主要行业；
2. 细分行业；
3. 分类依据；
4. 同行候选；
5. 同行可比理由；
6. 同行数据缺失说明。

注意：同行候选不等于最终估值可比公司。最终可比公司需要行业研究员和估值分析员继续筛选。

### 4.5 准备产业链和行业指标指针

信息收集员2应为行业研究员准备：

1. 上游原材料；
2. 中游生产、品牌、渠道；
3. 下游客户和消费场景；
4. 产业链利润池说明；
5. 价格指标；
6. 库存指标；
7. 动销或订单指标；
8. 政策监管指标；
9. 技术或产品趋势指标。

### 4.6 准备政策、新闻和市场数据缺口

如果没有实时数据源，信息收集员2必须明确标注缺失，而不是编造。

必须输出：

1. 已收集政策或新闻；
2. 推荐补采来源；
3. 未收集原因；
4. 对行业研究结论的影响；
5. 下一步补采请求。

### 4.7 维护证据和审计

每个行业输入包必须配套：

1. `industry_input_package.json`；
2. `industry_input_package.md`；
3. `evidence_table.json`；
4. `collection_audit.json`；
5. `validation_result.json`；
6. 总 manifest。

## 5. 当前已确认的硬规则

### 规则 1：唯一默认工作区

默认工作区是：

```text
industry_info_collector_scripts/collector_workspace
```

该目录下维护：

```text
collector_workspace/
  manifests/
    industry_packages.json
    industry_packages.csv
  packages/
    {stock_code}/{as_of_date}/
      industry_input_package.json
      industry_input_package.md
      evidence_table.json
      collection_audit.json
      validation_result.json
      industry_research_demo.json
      industry_research_demo.md
```

### 规则 2：不污染原财报收集员工作区

信息收集员2不得把行业信息包写入：

```text
info_collector_scripts/collector_workspace
```

该目录只属于原财报信息收集员。

### 规则 3：优先引用本地已有产物

信息收集员2应优先引用：

1. `info_collector_scripts/collector_workspace/manifests/cninfo_all_reports.json`；
2. `info_processor_scripts/processor_workspace/parsed_reports/.../content.json`；
3. `financial_analyst_scripts/analyst_workspace/reports/.../analyst_report.json`。

这些输入已经具备财报来源、页码、RAG、财务分析和限制条件，信息收集员2不应重复做低价值抓取。

### 规则 4：离线 seed 数据必须显式标注

当使用离线 seed 数据时，必须在证据中标注：

```text
source_type = curated_seed
```

并写明：

1. 该数据是演示或人工维护种子数据；
2. 不替代权威行业数据库；
3. 不代表实时状态；
4. 需要后续补采和验证。

### 规则 5：不伪造实时数据

如果没有可靠实时来源，以下字段必须标记为缺失或待补采：

1. 实时股价；
2. 市值；
3. PE / PB / PS / EV/EBITDA；
4. 高端白酒实时批价；
5. 渠道库存；
6. 动销数据；
7. 机构持仓；
8. 资金流向；
9. 实时新闻；
10. 最新政策原文。

### 规则 6：输出的是输入包，不是行业结论

信息收集员2可以说：

```text
当前已准备高端白酒批价、渠道库存、动销、消费税、食品安全监管等待跟踪指标。
```

不能说：

```text
高端白酒行业已经触底反转。
贵州茅台行业位置强，应该买入。
```

### 规则 7：每个输入包必须有审计文件

`collection_audit.json` 必须说明：

1. 本次是否离线；
2. 是否尝试网络请求；
3. 输入文件是否存在；
4. 输入文件 hash；
5. 数据可用性；
6. 缺失字段；
7. 限制条件；
8. 是否可交给行业研究员；
9. 推荐下一步补采。

### 规则 8：manifest 以 package_id 去重

行业信息包总清单使用：

```text
package_id = industry_input_{stock_code}_{fiscal_year}_{as_of_date}
```

同一天同一股票同一财年的输入包重跑时，更新同一条记录。

## 6. 标准输入

信息收集员2的输入包括四类。

### 6.1 必需输入

| 输入 | 含义 |
|---|---|
| `stock_code` | 股票代码 |
| `company_name` | 公司名称 |
| `fiscal_year` | 财报年度 |
| `as_of_date` | 信息包生成日期 |

### 6.2 推荐本地输入

| 输入文件 | 用途 |
|---|---|
| `analyst_report.json` | 提供财务分析员的公司画像、财务摘要、风险和限制条件 |
| `content.json` | 提供财报元数据、正文、表格、页码和业务分部原始证据 |
| `cninfo_all_reports.json` | 提供财报公告来源、PDF 地址和本地路径 |
| `industry_seed_data.json` | 提供离线行业分类、同行候选、产业链和待跟踪指标 |

### 6.3 可选增强输入

后续可扩展：

1. 行业协会数据；
2. 统计局数据；
3. 行情估值数据库；
4. 新闻数据库；
5. 政策数据库；
6. 专家访谈纪要；
7. 卖方研报摘要；
8. 同行年报和财务分析员报告。

```json
  "research_scope": {},

```

### 7.2 人类可读摘要

```text
industry_input_package.md
```

应说明：

1. 本次收集对象；
2. 使用的输入文件；
3. 公司画像；
4. 财务摘要；
5. 初始行业分类；
6. 同行候选；
7. 产业链和指标；
8. 缺失数据；
9. 下一步补采。

```text
evidence_table.json
```

每条证据至少包含：

1. `ref_id`；
2. `claim`；
3. `evidence`；
4. `source_type`；
5. `source_path`；
6. `source_detail`；
7. `reliability`；
8. `as_of_date`；

```text
collection_audit.json
```

### 7.5 总 manifest

```text
manifests/industry_packages.json

```

记录每个行业输入包的路径、行业分类、来源模式、输入质量和是否可用。

## 8. 标准工作流

### 工作流 A：离线单公司行业输入包生成

适用场景：

- 已有目标公司财报和财务分析员报告；
- 需要先打通行业研究员输入链路；
- 网络不可用或暂不接入实时行业数据。

步骤：

1. 输入股票代码、公司名称、财年和生成日期；
2. 自动定位或手动传入财务分析员报告；
3. 自动定位或手动传入 `content.json`；
4. 读取原财报 manifest；
5. 读取行业 seed 数据；
6. 生成 `industry_input_package.json`；
7. 生成 `industry_input_package.md`；
8. 生成证据表和收集审计；
9. 更新总 manifest；
10. 运行校验脚本；
11. 将结果交给行业研究员。

### 工作流 B：带外部数据增强的行业输入包生成

适用场景：

- 已接入行情、估值、行业协会、新闻和政策数据；
- 需要提高行业研究结论置信度。

步骤：

1. 先执行工作流 A；
2. 补充实时或准实时行业数据；
3. 对每个外部来源记录来源、日期、可靠性和限制；
4. 更新输入包和证据表；
5. 输出新的审计结论。

当前版本暂不默认执行工作流 B。

### 工作流 C：缺口补采请求

适用场景：

- 行业研究员发现输入包不足以支持某个行业判断；
- 需要补充价格、库存、政策、同行、估值或新闻数据。

步骤：

1. 行业研究员输出结构化缺口；
2. 信息收集员2把缺口转成补采任务；
3. 补采后更新输入包；
4. 保留旧版本限制条件和新版本变化说明。

## 9. 茅台离线样例口径

以贵州茅台 `600519` 为例，信息收集员2可以在离线模式下准备：

1. 公司名称、股票代码、交易所、报告期；
2. 年报公告来源和 PDF 路径；
3. 财务分析员报告路径；
4. 营业收入、归母净利润、扣非归母净利润、经营现金流；
5. 主营业务：茅台酒及系列酒生产销售；
6. 初始行业分类：白酒 / 高端白酒；
7. 同行候选：五粮液、泸州老窖、山西汾酒、洋河股份；
8. 产业链：高粱、小麦、包材、基酒产能、品牌运营、经销商、电商、团购、商务宴请和个人消费；
9. 待跟踪行业指标：高端白酒批价、渠道库存、动销、终端需求、消费税和监管；
10. 限制条件：未接入实时批价、库存、同行财务、估值、新闻和政策全文。

这些数据足以让行业研究员完成初步分析框架，但不足以形成高置信实时行业景气结论。

## 10. 质量控制

### 10.1 输入完整性

校验项：

1. 公司基础信息是否完整；
2. 财务分析员报告是否存在；
3. `content.json` 是否存在；
4. 财报 manifest 是否存在；
5. seed 数据是否存在；
6. 行业分类是否存在；
7. 同行候选是否至少 3 个；
8. 证据表是否非空；
9. 限制条件是否明确。

### 10.2 输出质量

合格输出必须满足：

1. 不输出最终行业结论；
2. 不输出投资建议；
3. 不伪造实时数据；
4. 每个关键字段有来源；
5. 缺失数据明确列出；
6. 可交给行业研究员继续分析。

### 10.3 置信度规则

输入质量分为：

- `high`：本地财务资料、权威行业数据、同行数据、政策数据和市场数据都较完整；
- `medium`：本地财务资料完整，行业分类和同行候选可用，但实时行业数据不足；
- `low`：只有 seed 数据或公司基础信息，缺少财务和行业证据。

茅台离线样例默认最高只能是 `medium`，因为没有实时批价、渠道库存、同行最新数据和估值快照。

## 11. 最终交付标准

一次合格的信息收集员2交付必须说明：

1. 本次收集对象；
2. 本次是离线模式还是网络增强模式；
3. 使用了哪些本地输入；
4. 使用了哪些 seed 或外部数据；
5. 输出了哪些文件；
6. 哪些数据可以直接给行业研究员使用；
7. 哪些数据只是研究指针；
8. 哪些关键数据缺失；
9. 当前输入包是否足以支持行业研究员做初步分析；
10. 当前输入包是否足以支持高置信行业结论。
